# Market Research Workflow

작성일: 2026-07-12

## 목적

in C의 시장조사는 창업오디션 제출을 위한 일회성 자료 수집이 아니라, 최소 인원으로
제품 전략과 사업 가설을 계속 업데이트하기 위한 운영 자산이다.

핵심은 큰 자동화부터 만드는 것이 아니다. 먼저 30분 안에 반복 가능한 수동 루틴을
만들고, 반복되는 복사, 표 업데이트, 체크리스트 생성만 자동화 후보로 올린다.

## 최소 인원 운영 원칙

- 한 번 조사한 수치는 출처, 기준연도, 조회일, 사용 위치를 함께 남긴다.
- 공식 통계와 표본 조사를 분리한다.
- 월간 루틴은 30분, 분기 루틴은 반나절 안에 끝나는 범위로 제한한다.
- 자동화는 비용 절감과 최신성 유지를 위한 보조 수단이며, 확인되지 않은 수치를
  자동으로 사업계획서 문장에 넣지 않는다.
- 개인정보, 인터뷰 원문, 연락처, 녹취, 미공개 협력자 정보는 레포에 저장하지 않는다.
- 외부 사이트 크롤링이나 scraping은 약관, robots, API 사용 조건 확인 전 구현하지 않는다.

## 월간 루틴

1. 이번 달 표본 공연 10건을 고른다.
2. 공연 규모, 가격대, 홍보 채널, 프리뷰/해설 여부를 기록한다.
3. 고정 키워드 그룹의 검색 추세를 확인한다.
4. 새로 눈에 띄는 대체재/콘텐츠 포맷을 1-3개만 메모한다.
5. 새 수치가 있으면 출처와 조회일을 남기고, 확정/표본/추정을 표시한다.
6. 제품 이슈에 반영할 내용이 있으면 관련 이슈에 코멘트하거나 후속 이슈로 분리한다.

## 분기 루틴

1. 공식 통계 업데이트 여부를 확인한다.
2. 공연장 규모/객석 수 데이터의 변경 여부를 확인한다.
3. 대체재 비교표를 갱신한다.
4. 교육/클래스/악보 도구 시장의 공개 가격과 포맷 변화를 확인한다.
5. 사업계획서, 피치덱, 제품 로드맵 문장 중 낡은 표현을 갱신한다.

## 자동화 가능한 자료원 후보

- 공개 API 또는 다운로드: KOPIS, 공공데이터 포털 후보 [확인 필요]
- 정적 공개 페이지: 공연장 일정, 예매처 공개 목록 [약관 확인 필요]
- 검색 트렌드 도구: 네이버 데이터랩, Google Trends, 키워드 도구 [확인 필요]
- 직접 관리 데이터: 공연장 규모 표, 표본 공연 URL, 대체재 카탈로그
- GitHub: 월간 체크리스트 issue template, Markdown 표 업데이트 스크립트

## 자동화하지 말아야 할 자료

- 인터뷰 원문, 연락처, 녹취, 실명
- 로그인 뒤에만 볼 수 있는 데이터
- 약관상 자동 수집이 제한된 페이지
- 포스터, 이미지, 해설 텍스트 원문
- SNS/커뮤니티 개인 게시글 원문과 개인정보

## 반영 흐름

`자료 수집 -> 출처 기록 -> 확정/표본/추정 분류 -> 제품 해석 -> GitHub 이슈 연결 -> 오디션/피치덱/제품 문서 반영`

확정 전에는 사업계획서 문장에 숫자를 넣지 않는다. 표본 조사는 반드시 표본이라고
표시한다.

제품 범위나 사업 가설로 옮길 때는
[`decision-log-template.md`](decision-log-template.md)의 형식으로 근거, 대안,
후속 이슈, 재검토 날짜를 남긴다.

## 월간 체크리스트 생성 설계 (#298)

초기에는 GitHub issue template 또는 이전 달 issue 복사를 사용한다. 로컬 스크립트는 같은
체크리스트를 2-3회 반복한 뒤에만 만든다.

| 후보 | 장점 | 제한 | 추천 단계 |
| --- | --- | --- | --- |
| GitHub issue template | 웹에서 바로 생성 가능, 이슈 연결 쉬움 | 템플릿 변경 이력 관리 필요 | 1차 |
| 이전 달 issue 복사 | 5분 안에 실행 가능 | 누락/오타 가능 | 1차 대체 |
| 로컬 스크립트 + `gh issue create` | 반복 생성 안정적 | 스크립트 유지보수 필요 | 2-3회 반복 뒤 |
| GitHub Actions | 완전 자동화 가능 | 권한/일정/알림 관리 필요 | 운영 부담이 확인된 뒤 |

### 월간 issue 제목

`YYYY-MM 시장조사 체크리스트`

### 월간 issue 본문 템플릿

```md
## 이번 달 범위

- [ ] 공연 홍보 표본 10건 기록 (#295)
- [ ] 키워드 그룹 추세 확인 (#296)
- [ ] 새 출처 후보를 카탈로그에 추가 (#292)
- [ ] 개인정보/약관/저작권 체크 (#300)

## 표본 요약

- 공연 표본:
- 키워드 변화:
- 제품 이슈로 연결할 내용:

## 완료 기준

- [ ] 표본과 출처에 조회일을 남겼다.
- [ ] 표본/추정/확정을 구분했다.
- [ ] 원문, 개인정보, 저작권 자료를 저장하지 않았다.
```

수동 생성 흐름은 `새 issue 만들기 -> 제목의 월만 바꾸기 -> 본문 템플릿 붙여넣기 -> 관련
이슈 링크 확인`으로 제한한다.

## 개인정보, 약관, 저작권 체크리스트 (#300)

이 체크리스트는 #291-#299의 모든 수동 조사와 자동화 후보 앞에 적용한다. 법률 판단이
필요하면 `[확인 필요]`로 남기고 구현 또는 제출 반영을 미룬다.

### 개인정보와 인터뷰

- [ ] 실명, 연락처, 이메일, 계정명, 단체 채팅방 정보가 레포에 들어가지 않는다.
- [ ] 인터뷰 원문, 녹취, 캡처 대신 익명 코드와 요약 인사이트만 남긴다.
- [ ] 현장조사 메모는 개인을 식별할 수 없도록 사건, 장소, 시간을 필요 이상으로 적지 않는다.
- [ ] 동의 범위가 불명확한 자료는 제품 문서나 제출 문장에 쓰지 않는다.

### 외부 사이트 약관/API/robots

- [ ] 출처 카탈로그에 URL, 접근 방식, 약관 확인 상태를 적었다.
- [ ] API나 다운로드가 있으면 scraping보다 우선한다.
- [ ] robots 또는 약관이 불명확하면 자동 수집을 구현하지 않는다.
- [ ] 로그인, 우회 접근, 세션 쿠키가 필요한 자료는 수집 대상에서 제외한다.

### 저작권과 원문 복제

- [ ] 공연 포스터, 이미지, 해설 텍스트, 악보, 번역문 원문을 레포에 복제하지 않는다.
- [ ] 상세페이지와 SNS 게시물은 링크, 짧은 요약, 메타데이터 중심으로 기록한다.
- [ ] 권리 상태가 불명확한 PDF, MusicXML, 악보 파일은 샘플 데이터로 저장하지 않는다.
- [ ] 사업계획서에는 출처 있는 수치와 직접 작성한 해석만 사용한다.

### 자동화 구현 전 승인 기준

| 기준 | 승인 조건 |
| --- | --- |
| 출처 적합성 | 출처 카탈로그에 등록되어 있고 사용 제한이 확인되어 있다 |
| 데이터 범위 | 개인정보, 원문 콘텐츠, 비공개 데이터가 없다 |
| 수집 방식 | API/다운로드/수동 입력이 우선이고 scraping은 약관 확인 뒤 별도 검토한다 |
| 검토 흐름 | 자동 반영 전 사람이 `git diff`와 출처를 확인한다 |
| 실패 대체 | 자동화 실패 시 수동 표본 조사로 대체할 수 있다 |
