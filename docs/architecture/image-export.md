# 이미지 출력 정책

## 목적

이미지 출력은 PDF 변환과 별개의 공유용 내보내기 기능이다. 사용자는 악보 전체
또는 페이지 단위 이미지를 문서, 웹, 메신저에 삽입할 수 있어야 한다.

## 1차 포맷

1차 구현 포맷은 PNG로 한다.

- SVG 원본 내보내기는 renderer 내부 SVG 구조와 폰트 의존성이 안정된 뒤 검토한다.
- PNG는 사용자가 바로 공유하기 쉽고, PDF보다 단일 이미지 삽입 흐름이 단순하다.
- PDF 변환과 동일한 화면 layout을 사용해 화면, PDF, 이미지가 서로 다른 악보처럼
  보이지 않게 한다.

## 출력 범위

1차 범위는 페이지 단위 전체 악보 이미지다.

- 선택 영역 또는 특정 마디만 내보내기는 후속 기능으로 둔다.
- 다중 페이지가 생기면 `파일명-page-1.png`, `파일명-page-2.png`처럼 페이지별
  파일을 생성한다.
- 현재 단성부 MVP에서는 한 페이지 또는 현재 layout 결과 전체를 먼저 지원한다.

## 렌더링 기준

- 배경: 기본값은 흰 배경이다.
- 투명 배경: 후속 옵션으로 검토한다.
- 해상도: 기본값은 화면 배율보다 선명한 2x scale이다.
- 여백: PDF 변환과 같은 page margin 정책을 사용한다.
- 폰트와 기보 요소: 화면 렌더링과 같은 VexFlow 결과를 사용한다.

## 저장 UX

- 메뉴 이름: `이미지로 내보내기`
- 기본 파일명: `${score.title}.png`를 안전한 파일명으로 정규화한다.
- 저장 위치: 사용자가 save dialog에서 직접 고른다.
- 취소: 조용히 취소하고 상태 메시지를 바꾸지 않는다.

## 실패 메시지

- 렌더링 대상을 찾지 못한 경우: `이미지로 만들 악보 화면을 찾을 수 없습니다.`
- 파일 저장 실패: `이미지 파일을 저장하지 못했습니다.`
- 미지원 범위 요청: `선택 영역 이미지는 아직 지원하지 않습니다.`

## 저작권과 표기

이미지 출력물에 in-C 워터마크를 강제하지 않는다. 사용자가 작성하거나 가져온
악보의 권리와 출처 표기는 사용자 책임으로 둔다. 다만 Compositions처럼 in-C가
제공하는 공개 악보 페이지에서는 출처와 라이선스를 페이지 metadata로 별도 관리한다.

## 후속 구현 조건

- PDF 변환과 이미지 출력 버튼이 UI에서 명확히 구분되어야 한다.
- 출력은 기존 score-core 의미를 변경하지 않는 순수 export여야 한다.
- renderer snapshot 테스트 또는 packaged smoke test로 빈 이미지가 생성되지 않는지
  확인해야 한다.
