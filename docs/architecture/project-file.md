# MusicXML 우선 저장 전략

이 문서는 in-C의 현재 파일 저장 방향을 정리한다. 2026-07-03 기준으로 in-C는
별도 전용 프로젝트 파일보다 MusicXML 가져오기·내보내기 흐름을 우선한다.

## 현재 결정

- 첫 공개와 단성부 MVP에서는 MusicXML을 주 작업 교환 포맷으로 사용한다.
- 앱 UI에서는 `저장`이라는 표현을 프로젝트 파일 의미로 미리 사용하지 않는다.
- 현재 파일 작업은 `MusicXML 가져오기`, `MusicXML 내보내기`, `PDF 변환`으로
  구분한다.
- `.incproj` 같은 전용 프로젝트 파일은 지금 구현하지 않는다.
- 전용 프로젝트 파일은 실제 사용자 문제나 기능 요구가 생기면 새 이슈로 다시
  검토한다.

## 판단 이유

현재 단계에서 전용 파일은 구체적인 사용자 불편에서 나온 요구가 아니다. 단성부
MVP의 핵심 사용 흐름은 다음 두 가지다.

- 작성한 악보를 MusicXML로 다른 도구와 주고받는다.
- 작성한 악보를 PDF로 읽고 공유한다.

MusicXML만으로 보존하기 어려운 in-C 고유 작업 상태가 늘어나기 전까지는 별도
프로젝트 포맷이 제품과 UI를 불필요하게 복잡하게 만들 수 있다.

## 사용자-facing 용어

- `MusicXML 가져오기`: 외부 MusicXML 파일을 in-C score로 변환한다.
- `MusicXML 내보내기`: 현재 악보를 외부 교환용 `.musicxml` 파일로 생성한다.
- `PDF 변환`: 현재 악보를 읽기/공유용 PDF로 생성한다.
- `저장`: 전용 프로젝트 파일을 구현하기 전까지 주요 버튼 이름으로 쓰지 않는다.

## 전용 프로젝트 파일을 다시 검토할 조건

다음 필요가 실제로 생기면 `.incproj` 또는 다른 전용 작업 포맷을 새 이슈로 검토한다.

- MusicXML로 보존하기 어려운 in-C 고유 편집 상태가 제품상 중요해질 때.
- 선택 범위, 입력 커서, 레이아웃 힌트, 앱 전용 metadata를 다시 열 때 복구해야 할 때.
- MusicXML 왕복 손실 때문에 사용자의 작업 원본을 별도로 보존해야 할 때.
- 자동저장과 복구가 MusicXML 기반 임시 저장만으로 부족해질 때.
- 웹 문서, 커뮤니티, 협업 기능에서 별도 document metadata가 필요해질 때.

## 자동저장과 복구

자동저장과 복구는 여전히 유효한 제품 과제다. 다만 전용 프로젝트 파일을 전제로
구현하지 않는다.

현재 방향에서는 자동저장을 검토할 때 다음 대안을 먼저 비교한다.

- MusicXML 임시 파일을 저장한다.
- score-core JSON snapshot을 앱 내부 복구용으로 저장한다.
- 사용자가 명시적으로 내보낸 MusicXML 파일은 조용히 덮어쓰지 않는다.

복구 UX는 사용자가 명시 저장한 파일을 자동으로 덮어쓰지 않아야 하며, 복구본을 열어
확인한 뒤 사용자가 다시 MusicXML로 내보내는 흐름을 기본값으로 둔다.
