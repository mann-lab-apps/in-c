# MusicXML 외부 샘플 후보와 반입 기준

이 문서는 다른 사보 도구가 내보낸 MusicXML을 호환성 검증에 쓰기 위한 후보와
판정 기준을 정리한다. 외부 파일 자체는 이 작업에서 내려받거나 저장소에 넣지 않는다.
현재 지원 범위는 [MusicXML MVP 지원 범위](../musicxml-mvp.md)를 기준으로 한다.

## 공통 시험 악보

도구별 차이를 비교할 때는 제3자의 악보를 복제하지 않고 검토자가 직접 만든 같은
악보를 사용한다.

- 단일 part, 단일 staff, 높은음자리표
- 4/4박자, C장조, 2마디
- 1마디: 4분음표 C4·D4·E4·F4
- 2마디: 2분쉼표와 2분음표 G4
- 제목: `Chromatics MusicXML smoke`, 작곡가: `mann-lab-apps QA`
- 압축하지 않은 `.musicxml` 또는 `.xml`

이 악보는 현재 MVP 범위의 회귀 샘플이다. 여러 part, 여러 voice, `backup`,
`forward`, 압축 `.mxl`처럼 현재 지원하지 않는 구조는 별도의 탐색 샘플로 기록하며
통과 회귀 fixture와 섞지 않는다.

## 후보 목록

| 우선순위 | 후보 | 형식·버전 확인 근거 | 대표 확인 기능 | 권리·반입 상태 | 다음 행동 |
| --- | --- | --- | --- | --- | --- |
| 1 | MuseScore Studio에서 공통 시험 악보 내보내기 | [공식 handbook](https://handbook.musescore.org/en_gb/file-management/file-export)은 MuseScore 4의 MusicXML 4.0 내보내기를 설명한다. | 단일 part, 음표·쉼표, 제목, 작곡가 | 검토자가 직접 만든 악보만 사용하면 원곡 권리 위험을 줄일 수 있다. 프로그램 버전과 export 설정 증빙은 아직 필요하다. `보류` | 설치된 버전과 uncompressed MusicXML 설정을 기록한 뒤 수동 생성한다. |
| 2 | Dorico SE에서 공통 시험 악보 내보내기 | [공식 도움말](https://www.steinberg.help/r/dorico-se/6.1/en/dorico/topics/project_file_handling/project_file_handling_export_musicxml_dialog_r.html)은 압축·비압축 MusicXML 선택을 설명한다. | 같은 악보를 다른 exporter로 비교 | 검토자가 직접 만든 악보만 사용한다. 제품 이용 조건과 생성 파일 재배포 가능 여부 확인 전에는 `보류` | 제품 버전, export 설정, 이용 조건 확인자를 기록한다. |
| 3 | W3C MusicXML 4.0 `Hello World` 예제 | [공식 예제](https://www.w3.org/2021/06/musicxml40/tutorial/hello-world/)는 단일 part와 온음표 한 개의 MusicXML 4.0 구조를 공개한다. | 최소 `score-partwise` parser 확인 | 명세의 공개 열람과 저장소 재배포 허용은 같은 뜻이 아니다. 적용 조건 확인 전에는 `보류` | W3C 적용 조건과 원문 재배포 허용 범위를 확인한다. 허용이 불명확하면 URL 참조만 유지한다. |

후보 표의 `보류`는 호환성이 나쁘다는 뜻이 아니다. 파일 반입에 필요한 증빙이 아직
없다는 뜻이다. 로그인, 유료 계정, 약관 우회 또는 자동 수집이 필요한 후보는 추가하지
않는다.

## 수용·보류·거부 기준

### 수용

다음을 모두 만족할 때만 저장소 fixture 후보로 수용한다.

- 공식 export 문서 또는 파일의 `software` 정보로 도구와 버전을 확인할 수 있다.
- MusicXML 형식과 버전, 압축 여부를 기록했다.
- 원본 URL, 취득일, 라이선스 또는 직접 작성 증빙을 남겼다.
- 저장소 재배포가 허용됨을 확인했다.
- 예상 결과가 현재 MVP의 지원 또는 미지원 범위와 연결되어 있다.
- 파일에 개인정보, 로컬 경로, 계정명 같은 불필요한 정보가 없다.
- SHA-256 해시로 검토한 파일과 반입 파일이 같은지 확인할 수 있다.

### 보류

출처는 알지만 라이선스, 재배포 조건, 생성 버전 또는 예상 결과 중 하나가 확인되지
않으면 보류한다. 담당자와 확인할 항목을 기록하고 파일은 저장소에 넣지 않는다.

### 거부

다음 중 하나라도 해당하면 거부한다.

- 권리 상태나 출처가 불명확하다.
- 로그인 우회, scraping 또는 비공개 링크가 필요하다.
- 제3자의 저작권 악보 원문이나 개인정보를 포함한다.
- 손상 파일처럼 목적은 있으나 재현 조건과 기대 오류를 설명할 수 없다.
- 현재 지원 범위 회귀 샘플로 표시했지만 여러 part·voice 또는 압축 `.mxl`만 제공한다.

## 호환성 판정

파일의 반입 판정과 앱의 호환성 판정은 따로 기록한다.

- `회귀 통과`: 현재 MVP 범위이고 가져오기, 내보내기, 다시 가져오기가 성공한다.
- `예상 거부`: 미지원 구조가 있으며 앱이 조용히 내용을 버리지 않고 오류를 알린다.
- `실패`: 지원 범위인데 열리지 않거나, round-trip 뒤 음악 의미가 달라진다.
- `미실행`: 파일 반입 전이거나 검증 근거가 없다.

## 후보 검토 기록 양식

```markdown
## 외부 MusicXML 후보 검토: YYYY-MM-DD

- 후보/내보내기 도구:
- 도구 버전:
- 원본 URL 또는 직접 작성 증빙:
- 취득일:
- MusicXML 형식·버전/압축 여부:
- 라이선스·재배포 근거:
- SHA-256:
- 대표 기능:
- 현재 지원 범위 연결:
- 예상 결과: 회귀 통과 / 예상 거부
- 반입 판정: 수용 / 보류 / 거부
- 판정 이유:
- 검토자:
- 관련 이슈:
```

수용된 파일의 실제 다운로드, fixture 추가, parser 실행과 결과 기록은 별도 이슈나
PR에서 수행한다. 외부 파일을 추가할 때는 파일 가까이에 출처와 라이선스 기록도 함께
둔다.
