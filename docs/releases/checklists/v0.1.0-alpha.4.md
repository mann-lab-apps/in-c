# v0.1.0-alpha.4 릴리즈 체크리스트

릴리즈 후보를 배포하기 전에 이 문서의 항목을 모두 확인한다. 항목은 일부러
중복될 수 있다. 자동 테스트가 통과했더라도 Codex 점검과 사람 점검에서 같은
기능을 다시 확인하면 릴리즈 위험을 줄일 수 있다.

## 릴리즈 정보

- 버전: `0.1.0-alpha.4`
- 이전 태그: `v0.1.0-alpha.3`
- 릴리즈 태그: `v0.1.0-alpha.4`
- 릴리즈 날짜: `2026-07-11`
- 점검 위치: `release/0.1.0-alpha.4`
- 담당: Codex, 사용자
- 릴리즈 판정: `대기`

## 결과 표기

- `통과`: 확인했고 릴리즈를 막지 않는다.
- `실패`: 릴리즈를 막는 문제가 확인됐다.
- `대기`: 아직 실행할 수 없거나 순서상 나중에 확인한다.
- `차단`: 외부 조건이나 결정이 필요해 진행하지 못한다.

하나라도 `실패`가 남아 있으면 태그를 만들지 않는다. `대기` 항목은 릴리즈 순서상
나중에 확인해야 하는 항목인지, 실제로 릴리즈를 막는 항목인지 판단을 적는다.

  이번 점검에서는 `npm run package:dir`까지만 실행했다.

  는 통과했다.
- `npm run package:dir`은 macOS Developer ID 인증서가 없어 signing만 skip했고,
  directory package 생성과 `npm run verify:package`는 통과했다.

## Codex가 사전 점검

Codex는 테스트 결과와 저장소 상태를 기준으로 릴리즈 위험을 확인한다.

- [x] `git log v{previous}..HEAD --oneline`으로 릴리즈 범위를 요약했다.
- [x] `git diff --name-only v{previous}..HEAD`로 변경 범위를 확인했다.
- [x] 새 기능 또는 큰 변경에 대응하는 테스트가 추가되어 있는지 확인했다.
  - MusicXML, editor state, keyboard input, tuplet feedback, notation layout,
    visual state, playback timeline, acceptance traceability 테스트를 확인했다.
- [x] `TODO`, `FIXME`, 임시 fallback, 오래된 버전 문자열을 검색했다.
  - 릴리즈 준비 전 상태로 `0.1.0-alpha.3` 문자열이 남아 있다.
- [x] `docs/distribution.md`의 배포 절차가 현재 workflow와 맞는지 확인했다.
  - release checklist 링크는 최신 경로다.
  - `docs/architecture/branch-strategy.md`와 브랜치 운영 설명이 맞도록
    `docs/distribution.md`를 갱신했다.
- [x] `README.md`의 검증, 패키징, 웹사이트 안내가 현재 스크립트와 맞는지 확인했다.

  확인했다.
  - native artifact 생성 후 `npm run verify:artifacts`로 재확인한다.
- [x] Release notes 초안이 이전 릴리즈 이후 변경을 과장하거나 누락하지 않는지
  확인했다.

    MusicXML 안정화, playback, Columns/Compositions 사이트, acceptance/test
    coverage, 릴리즈 체크리스트.
