# in C 제품 로드맵

작성일: 2026-07-11

이 문서는 in C의 제품 방향을 외부 지원사업, 피치덱, 제품 설명에 재사용할 수
있도록 정리한다. 세부 작업 단위와 우선순위는 GitHub 이슈와 feature map에서
관리한다.

## 현재 기준

in C의 제품 목표는 클래식에 대해 모르는 것을 숨기지 않고, 각자가 들은 것과 듣지
못한 것을 솔직하게 나눌 수 있는 소통 공간을 만드는 것이다.

현재 레포에서 가장 많이 구현된 제품은 Chromatics 악보 편집기지만, 2026년 7월의
집중 구현 범위는 Columns, 단선율 악보 라이브러리, 작품 페이지 관계 구조, 공연
프리뷰, Communities/Connects, Classes까지 앞당겨 잡는다. 다만 문서에서는
이미 구현된 기능과 7월 내 구현 목표를 구분한다.

현재 결정은 in C 브랜드 아래 웹을 공개 탐색과 공유 기준 표면으로, 데스크탑 앱을
깊은 악보 작업 표면으로 두는 것이다. Chromatics는 별도 브랜드가 아니라 그 악보
창작/편집 표면 이름이다. 모바일 앱은 감상과 소통의 일상 표면 후보로 남겨 둔다.
표면은 나누지만 같은 행동 이름이 표면마다 다르게 동작하지 않도록 한다.

## 2026년 7월 집중 구현 범위

7월 목표는 이전 청사진을 실제 사용자가 만질 수 있는 최소 제품군으로 앞당기는
것이다. 모든 영역을 완성형으로 만들기보다, 작품과 감상 질문을 중심으로 서로
연결되는 첫 구조를 구현한다.

### 1단계: Columns MVP

목표는 솔직한 클래식 대화의 톤과 철학을 가장 빨리 현실에 보여주는 것이다.

- 글 목록과 상세
- 관리자 작성, 수정, 삭제 [2026년 7월 목표]
- Markdown 또는 리치 텍스트 본문
- 공개, 비공개, 초안 상태
- slug, 제목, 요약, 대표 이미지, 카테고리, 태그
- 관련 작품, 작곡가, 공연, 악보, Columns 연결
- SEO 메타데이터
- 공유 기능
- 기본 피드백 기능 [2026년 7월 목표]
- GA 이벤트

### 2단계: 단선율 민요 라이브러리

목표는 검색 유입과 작품 탐색을 위한 안전한 악보 자산을 쌓는 것이다.

- 저작권 안전한 한국 및 세계 민요 선별
- 직접 MusicXML 입력
- Chromatics에서 열기와 PDF 변환 안내
- 웹 미리보기
- 메타데이터와 작품 페이지 연결
- 관련 Columns 연결
- GA 다운로드 이벤트

### 3단계: 작품 페이지와 관계 구조

목표는 작품을 중심으로 작곡가, 악보, Columns, 공연이 연결되는 허브를
만드는 것이다.

- 작품
- 작곡가와 작사가
- 악보
- Columns
- 공연
- 관련 인물 메타데이터
- 저작권 상태와 출처

### 4단계: 공연 배너와 등록 후보

목표는 실제 예정 공연과 작품을 연결하되, 독립 공연 탭이 아니라 홈 배너와
작품 상세의 배너 후보 관계로 먼저 검증하는 것이다. 전문가 그룹에게는 공연 홍보와
정보교류가 포스터, SNS, 지인 네트워크에 분산되는 부담을 줄이고, 공연이
작품/Columns 관계 속에서 발견되게 하는 기반을 만든다.

- 배너 슬롯과 등록 후보 UI [2026년 7월 목표]
- 실제 예정 공연과 작품 연결 후보 [2026년 7월 목표]
- 짧은 감상 가이드 [2026년 7월 목표]
- Columns와 유튜브 콘텐츠 재사용 [2026년 7월 목표]
- 공연 후 피드백 수집 [2026년 7월 목표]

### 5단계: Communities 또는 Connects

목표는 작품과 공연 중심 대화를 작게 검증하는 것이다.

- 작품별 감상과 질문 [2026년 7월 목표]
- 공연 후 관객 피드백 [2026년 7월 목표]
- 연주자와 관객의 소통 [2026년 7월 목표]
- 비전문가가 깊어진 취향을 공유하고 전문가가 관객 질문을 만나는 대화 구조 [2026년 7월 목표]
- 댓글, 컬렉션, 프로필, 그룹 중 최소 기능 선택 [확인 필요]

### 6단계: 사람/프로필 표면

목표는 작곡가, 연주자, 지휘자, 기획자 등 클래식 생태계 참여자를 어떤 방식으로 보여줄지
커뮤니티 기능 고도화 이후 다시 결정하는 것이다. 2026년 7월 공개 사이트에서는 별도
Creators 탭을 만들지 않고 공연·클래스·작품 안의 관련 인물 메타데이터로만 둔다.

- 공개 프로필/팔로우/관객 소통은 커뮤니티 모델과 함께 재검토 [확인 필요]
- 관련 인물 메타데이터는 공연·클래스·작품 안에서만 보조 정보로 유지

### 7단계: Community

목표는 감상 질문, 학습 후보, 향후 대화 흐름을 작품과 Columns 관계 속에서 탐색할
수 있게 하는 것이다. 클래스 내용은 Community 안에서 먼저 검증하고, 양이 많아지면
하위 탭으로 분리한다.

- Community 안의 학습 후보 목록과 상세 [2026년 7월 목표]
- 관련 인물 표시는 보조 메타데이터로만 유지
- 관련 작품과 Columns 연결 [2026년 7월 목표]
- 신청/결제 흐름은 7월 이후 검토 [확인 필요]

### 8단계: Chromatics 강화

목표는 악보 라이브러리와 커뮤니티를 창작 도구로 연결하는 것이다.

- 악보 라이브러리와 직접 연결
- MusicXML 열기와 수정
- Chromatics 화면에서 PDF 변환
- 저장과 공유
- 사용자 창작물 게시 [2026년 7월 목표]
- 사람/프로필 공개 표면은 커뮤니티 고도화 이후 재검토

### 9단계: 제품 표면 역할 결정

목표는 웹, 데스크탑 앱, 모바일 후보가 같은 기능을 무리하게 복제하지 않으면서도
사용자가 in C 안에서 같은 규칙으로 이해할 수 있게 하는 것이다.

- 웹: 공개 탐색, 작품/악보 상세, Columns, 공연, 커뮤니티의 기준 URL.
  장기적으로는 브라우저 편집, 링크 공유, 협업 편집까지 확장
- 데스크탑 앱: Chromatics 악보 창작/편집 표면, MusicXML, PDF 변환, 오프라인 편집
- 모바일 앱: Columns 읽기, 공연 전후 알림, 짧은 감상 기록, 질문과 댓글
- 웹/Chromatics 사이 행동 이름과 결과 일관성 유지
- 모바일 앱 MVP 범위 결정 [확인 필요]

다음 기능은 7월 최소 제품군이 실제로 연결된 뒤 검토한다.

- 대규모 SNS형 피드
- Figma처럼 브라우저에서 열고, 저장하고, 링크로 공유하는 웹 편집 표면

- 다성부와 여러 staff/part의 전문 사보 기능
- 코드 심벌과 가사
- 고급 페이지 레이아웃
- AI 악보 인식 또는 스캔 기능
- 유료 결제와 구독 기능
- Everyone's In C 실제 구현

## 로드맵 리스크

- Chromatics만 강조하면 in C가 단순 사보 프로그램으로 오해될 수 있다.
- 웹, 데스크탑, 모바일을 모두 같은 범위로 구현하려 하면 최소 인원 운영 원칙과
  충돌할 수 있다.
- Columns가 실제 발행과 사용자 반응 없이 구조만 커지면 철학이 제품 경험으로
  전달되지 않는다.
- 7월 구현 범위가 넓어졌기 때문에 각 영역이 얕은 껍데기로 끝날 수 있다.
- 커뮤니티 기능은 대화할 콘텐츠와 사용자를 함께 확보하지 않으면 빈 공간이 될 수
  있다.
- MusicXML 호환 범위를 넓힐수록 지원하지 않는 요소의 안내와 손실 방지 정책이
  중요해진다.
- 저작권 상태가 불명확한 자료를 성급하게 공개하면 신뢰를 잃을 수 있다.
- 사업화 가설은 아직 사용자 인터뷰와 베타 사용 데이터로 검증해야 한다.
