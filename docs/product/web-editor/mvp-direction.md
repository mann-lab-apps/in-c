# 웹 기반 악보 편집기 제품 방향

## 제품 역할

웹 편집기는 데스크톱 앱을 즉시 대체하지 않는다. 다만 장기 방향은 Figma처럼
브라우저에서 열고, 저장하고, 링크로 공유하며, 점차 함께 편집할 수 있는 서비스형
악보 편집 표면으로 성장하는 것이다.

MVP에서는 설치 없이 악보를 열고, 가볍게 수정하고, 링크로 공유하는 표면으로
시작한다. 고급 입력과 오프라인 작업은 당분간 데스크톱 앱의 Chromatics 편집 표면이
계속 맡는다.

## MVP 범위

- MusicXML 열기
- 단선율 악보 보기와 기본 편집
- 계정 기반 저장
- 공유 링크 생성
- 읽기 전용 공개 보기
- 데스크톱 앱에서 이어 열기
- 댓글은 범위 밖으로 두고 피드백 폼만 검토

## 데스크톱 앱과의 차이

| 영역 | 데스크톱 앱 | 웹 편집기 MVP |
| --- | --- | --- |
| 파일 | 로컬 MusicXML 중심 | API 저장 문서 중심 |
| 오프라인 | 필수 | 필수 아님 |
| 입력 | 키보드 중심 고속 편집 | 기본 수정과 확인 |
| 공유 | 파일 전달 | 링크와 권한 |
| 커뮤니티 | 얕은 진입점 | 게시와 공개 보기 연결 |

## 협업 단계

실시간 협업은 MVP 이후로 미룬다. 첫 단계는 비동기 저장, 공유 링크, 읽기 전용
권한을 구현하고, 이후 댓글, 버전 히스토리, 동시 편집 순서로 확장한다. 목표는
악보 파일을 주고받는 흐름보다 URL 하나로 최신 상태를 확인하고 함께 다듬는 흐름을
만드는 것이다.

## 렌더링 재사용

현재 renderer의 score model, MusicXML parse/serialize, notation preview는 웹
편집기에서도 재사용 후보이다. 단, Electron 전용 file dialog와 local storage
복구 로직은 웹 계정 저장으로 분리해야 한다.

## IA 초안

- 문서 목록: 최근 악보, 내 악보, 공유받은 악보
- 편집 화면: 악보, 속성 패널, 저장 상태, 공유 버튼
- 공유 모달: 공개 범위, 링크 복사, 권한
- 공개 보기: 작품/Creator/Columns 연결

## 후속 구현

- 웹 편집기 scaffold 위치 결정
- 공통 score package 분리 가능성 검토
- 저장 API와 문서 권한 contract
- 브라우저 렌더링 smoke test

## 전환 판단

웹 서비스형 편집기는 단기 대체 프로젝트가 아니라 장기 기본 편집 표면 후보이다.
다음 조건이 보이면 MVP scaffold를 별도 이슈로 만든다.

- Compositions에서 사용자가 MusicXML을 열어본 뒤 저장/공유를 요구한다.
- Supabase project와 문서 권한 schema가 준비된다.
- 공개 보기 URL, 공유 링크, 읽기 전용 권한이 제품 가치를 만든다.
- Electron file dialog와 autosave에 의존하지 않는 browser storage/fetch adapter가
  필요하다.

단기에는 웹 공개 탐색과 데스크톱 편집 표면을 함께 유지한다. 웹 편집기 MVP는 계정
저장, 공유 링크, 읽기 전용 공개 보기부터 시작하고, 고급 입력은 데스크톱 앱의
역할로 둔다. 이후 사용자가 웹에서 편집을 계속 이어가길 원하면 브라우저 편집 범위를
점진적으로 넓힌다.
