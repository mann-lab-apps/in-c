# 피드백 데이터 계약과 보관 기준

이 문서는 Columns의 선택형 피드백을 GA4로 확인하는 현재 흐름과, 향후
`feedback_events` 테이블에 저장할 때 지켜야 할 최소 규칙을 정한다. 자유 입력과
개인 식별 정보는 수집하지 않는다.

## 인수 기준

- 허용 필드와 각 필드의 값 범위를 한곳에서 확인할 수 있다.
- 화면에 있는 세 선택지만 `answer`로 저장한다.
- 익명 사용자는 응답을 추가할 수 있지만 읽거나 수정할 수 없다.
- 원본 응답의 보관 기간, 삭제 시점, 관리자 조회 범위가 명확하다.
- 현재 GA4 이벤트와 향후 `feedback_events` 저장을 구분한다.

## 현재 수집 범위

현재 공개 사이트는 별도 데이터베이스에 피드백을 저장하지 않는다.
`feedback_submit` 이벤트를 GA4로 보내 운영자가 질문별 응답 수를 확인한다. GA4
payload는 [분석 이벤트 규칙](analytics-events.md)의 허용 목록을 따른다.

향후 Supabase 저장을 승인하면 다음 필드만 `feedback_events`에 기록한다.

| 필드 | 형식과 허용 값 | 용도 |
| --- | --- | --- |
| `id` | 서버가 만든 UUID | 응답 중복 확인과 운영 기록 |
| `content_type` | 현재는 `column`만 허용 | 질문이 놓인 제품 표면 |
| `content_slug` | 공개 콘텐츠에 존재하는 slug | 응답 대상 콘텐츠 |
| `answer` | 아래 세 값 중 하나 | 사용자가 고른 답 |
| `created_at` | 서버가 기록한 UTC 시각 | 보관 기간 계산 |

`answer` 허용 값은 화면의 `site/columns.js`와 같아야 한다.

| 값 | 사용자에게 보이는 문구 |
| --- | --- |
| `heard_something` | 오늘 들을 지점이 생겼어요 |
| `still_confusing` | 아직 어렵게 느껴져요 |
| `want_next_question` | 다음 질문이 궁금해요 |

클라이언트가 보낸 `id`나 `created_at`은 받지 않는다. 허용 목록에 없는 필드와 값은
저장하지 않고 요청을 거절한다.

## 수집 금지 항목

- 이름, 이메일, 전화번호, 주소, 계정 ID, IP 주소
- 자유 입력 문장, 문의 내용, 악보 내용이나 비공개 파일 정보
- 브라우저 지문, 광고 식별자, 다른 서비스의 사용자 ID
- 관리자 메모나 응답자를 다시 알아낼 수 있는 조합 정보

새 질문이나 선택지를 추가할 때는 화면, 이 문서, 분석 이벤트 규칙, 공개 privacy
안내를 같은 변경에서 함께 검토한다. 자유 입력이 필요해지면 이 계약에 덧붙이지 않고
별도 개인정보 검토와 승인 이슈로 분리한다.

## 접근과 관리자 조회

- 익명 사용자는 허용된 응답을 추가할 수만 있다.
- 익명 사용자와 공개 API는 원본 응답을 조회, 수정, 삭제할 수 없다.
- 관리자 조회는 승인된 운영 담당자에게만 허용한다.
- 관리자는 기간, `content_type`, `content_slug`, `answer`별 개수만 우선 확인한다.
- 개별 원본 조회와 export는 오류 조사나 복구 확인에 필요한 경우로 제한하고 운영
  기록에 목적과 날짜를 남긴다.

Supabase 적용 시 행 단위 접근 정책(RLS)은 anon insert와 관리자 select만 허용한다.
update는 허용하지 않으며, delete는 보관 정책을 실행하는 관리자 작업에서만 허용한다.

## 보관과 삭제

- `feedback_events` 원본은 `created_at`부터 **90일** 보관한다.
- 운영 작업은 하루 한 번 90일이 지난 원본을 삭제하고, 실패하면 담당자에게 알린다.
- 주간·월간 판단에는 개인을 구분하지 않는 선택지별 합계만 사용한다. 원본 삭제 뒤에도
  합계를 남길 수 있지만 5건 미만 표본은 외부 공유하지 않는다.
- 비공개 백업의 원본 피드백은 일별 백업 만료 기준에 따라 최대 30일 더 남을 수 있다.
  만료된 백업은 복구 성공 기록을 확인한 뒤 삭제한다.
- 직접 식별자를 수집하지 않으므로 특정 응답자를 찾아 삭제하는 기능은 제공하지 않는다.
  식별 가능한 계정 정보와 연결하게 되면 수집을 시작하기 전에 삭제 요청 절차와 privacy
  안내를 다시 승인한다.

운영 시작 전에는 90일 보관, 백업 만료, 삭제 작업 담당자를 공개 privacy 안내에
반영하고 확인한다. 승인 전에는 Supabase project, 테이블, 예약 삭제 작업을 만들지 않는다.

## 실패 가능한 검증 기준

- 허용 필드 외의 값이 저장 가능하면 실패다.
- `answer`가 세 값과 다르거나 화면 선택지와 어긋나면 실패다.
- 직접 식별자나 자유 입력을 payload 또는 테이블에 추가하면 실패다.
- 익명 사용자가 원본 응답을 읽거나 수정할 수 있으면 실패다.
- 90일이 지난 원본이 삭제 대상이 아니거나 백업 만료 기준이 없으면 실패다.
- 현재 GA4 수집과 아직 승인되지 않은 Supabase 저장을 같은 운영 상태로 설명하면 실패다.
- `npm run verify:analytics` 또는 `git diff --check`가 실패하면 완료로 보지 않는다.
