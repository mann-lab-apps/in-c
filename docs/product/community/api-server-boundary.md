# 커뮤니티 API 서버 경계와 인증 모델

## 결정

커뮤니티 기능은 Electron 앱에 직접 포함하지 않고 `in-C API`가 소유한다.
Electron 앱, 웹 사이트, 향후 웹 편집기는 같은 API를 호출하되 각 표면은
자기 화면 상태와 로컬 편집 경험만 책임진다.

## 서버 책임

- 사용자 계정, 세션, 프로필, 역할을 관리한다.
- 악보 게시물, 공개 범위, 라이선스, 신고 상태를 저장한다.
- 연주회 홍보, 검수 상태, 결제 기록, 노출 기간을 저장한다.
- 배너와 추천 콘텐츠처럼 여러 표면에서 공유되는 운영 콘텐츠를 제공한다.
- 관리자 검수, 블라인드, 반려 사유, 감사 로그를 남긴다.

## 클라이언트 책임

- Electron 앱은 악보 편집과 로컬 파일 저장을 우선한다.
- 앱은 커뮤니티 게시, 홍보, 계정 상태를 얕은 진입점으로만 제공한다.
- 웹 사이트는 탐색, 공개 콘텐츠, 홍보 랜딩, 로그인 진입을 맡는다.
- 웹 편집기는 API 계정으로 저장, 공유, 권한 확인을 수행한다.

## 인증 방향

MVP는 이메일 매직링크를 기본안으로 둔다. 비밀번호 저장 부담을 줄이고,
데스크톱 앱과 웹 브라우저가 같은 이메일 계정을 공유하기 쉽기 때문이다.
소셜 로그인은 확장 후보로 남긴다.

필수 계정 필드는 다음과 같다.

| 필드 | 설명 |
| --- | --- |
| `id` | API 내부 사용자 식별자 |
| `email` | 로그인과 소유권 확인 기준 |
| `displayName` | 공개 프로필 이름 |
| `profileImageUrl` | 선택 프로필 이미지 |
| `role` | 기본 권한 역할 |
| `createdAt` | 계정 생성 시각 |

## 초기 데이터 모델

- `User`: 계정, 이메일, 역할, 상태
- `Profile`: 공개 이름, 소개, 링크, 소속
- `ScorePost`: 악보 게시물, MusicXML/PDF 파일 참조, 라이선스
- `CommunityPost`: 공지, 질문, 일반 게시물
- `ConcertPromotion`: 공연 홍보 등록 정보와 노출 상태
- `PaymentRecord`: 결제 제공자, 금액, 영수증, 환불 상태
- `ModerationCase`: 신고, 검수, 블라인드, 반려 이력

## 리스크

- 개인정보 보관 범위는 이메일과 결제 식별자 중심으로 최소화한다.
- 결제와 홍보 검수는 API에서 원자적으로 연결해야 한다.
- Electron 앱의 오프라인 사용성은 로그인 실패와 무관하게 유지한다.
- 관리자 기능은 별도 권한과 감사 로그 없이는 공개 운영에 쓰지 않는다.

## 후속 구현 이슈

- API 서버 scaffold와 배포 환경 구성
- 인증 API와 세션 저장소 구현
- 역할/권한 policy middleware 구현
- 커뮤니티 게시물과 연주회 홍보 CRUD 구현
- 관리자 검수 화면과 운영 로그 구현
