# App, Web, and API Direction

작성일: 2026-07-13

## 목적

in C 데스크탑 앱의 Chromatics 편집 표면, in C 웹 사이트, 향후 웹 편집기, API 서버가
서로 다른 속도로 발전해도 제품 방향이 흩어지지 않도록 1차 역할을 정리한다. 표면
결정은 #301과 `surface-strategy.md`를 기준으로 하며, 이 문서는 구현 우선순위를
나누기 위한 운영 기준이다.

## 표면별 1차 역할

| 표면 | 1차 역할 | 먼저 구현할 것 | 뒤로 미룰 것 |
| --- | --- | --- | --- |
| 데스크탑 앱 | Chromatics 편집 표면으로 로컬 악보 작성과 파일 작업 | MusicXML 저장/가져오기, PDF 변환, 시스템 상태 패널 | 계정 기반 협업, 커뮤니티 피드 |
| in C 웹 사이트 | 공개 허브와 검색 유입 | Columns, Compositions, 다운로드, 제품 설명 | 복잡한 악보 편집 |
| 웹 편집기 | Figma처럼 브라우저에서 저장·공유·협업하는 장기 편집 표면 후보 | Open in Chromatics 흐름 검증, 공유 가능한 작품 페이지 | 네이티브 앱 수준 파일 관리 |
| API 서버 | 계정, 저장, 커뮤니티, 홍보 데이터의 경계 | 인증/세션 계획, 콘텐츠/배너 계약, 이벤트 계측 | 실시간 협업, 결제 자동화 |

표면마다 기능 범위는 달라도 `열기`, `MusicXML 다운로드`, `PDF 변환`, `공유` 같은
행동 이름은 같은 결과를 기대할 수 있게 유지한다.

## 앱의 얕은 연동

- 계정 상태와 로그인 유도 문구
- 커뮤니티 알림 또는 웹 기능 안내 배너
- 연주회 홍보/Classes로 가는 외부 링크
- 다운로드와 업데이트 안내
- 사이트에서 열 수 있는 작품/공연 링크

## 웹의 깊은 연동

- 작품 페이지와 악보 원본의 관계 허브
- 웹 기반 악보 편집 후보. 장기적으로는 설치 없이 악보를 열고, 저장하고, 링크로
  공유하며, 점차 협업 편집까지 확장한다.
- 공유 가능한 Columns/Compositions/Community URL과 공연 배너 후보 링크
- 계정 기반 저장, 게시, 협업은 API 서버 결정 뒤 구현

## 기능별 우선순위

| 기능 | 우선 표면 | 이유 | 관련 이슈 |
| --- | --- | --- | --- |
| MusicXML 저장/열기 | Electron 앱 | 로컬 파일과 릴리즈 품질의 핵심 | #230, #240 |
| Open in Chromatics | 웹 사이트 -> 앱/웹 편집기 | Compositions와 창작 도구 연결 | #275 |
| 홍보 배너 | 데스크탑 앱 + in C 웹 사이트 | 새 표면을 얕게 소개 | #258, #259 |
| 계정/세션 | API 서버 + 웹 | 커뮤니티와 저장 기능의 기반 | #260-#263 |
| 공연 배너/Community | 웹 사이트 우선 | 공개 URL과 검색 유입이 중요 | #272, #274 |
