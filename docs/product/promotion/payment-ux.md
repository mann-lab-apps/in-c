# 연주회 홍보 결제 UX와 환불 흐름

## 결제 전 화면

결제 전에는 사용자가 구매하는 항목을 한 화면에서 확인해야 한다.

- 노출 위치와 예시
- 노출 기간
- 검수 필요 여부
- 결제 금액과 부가세 표시 방식
- 환불 가능 조건
- 승인 거절 시 수정 요청 또는 환불 처리 방식

가격과 수수료는 외부 결제 정책에 따라 바뀔 수 있으므로 이 문서는 금액을
확정하지 않는다.

## 상태표

| 상태 | 사용자 표시 | 운영 처리 |
| --- | --- | --- |
| `checkoutStarted` | 결제 진행 중 | 결제 세션 생성 |
| `paid` | 결제 완료, 검수 대기 | 검수 큐에 등록 |
| `paymentFailed` | 결제 실패 | 재시도 안내 |
| `paymentCanceled` | 결제 취소 | 초안 상태 유지 |
| `reviewRejected` | 수정 필요 또는 환불 가능 | 반려 사유 기록 |
| `refunded` | 환불 완료 | 노출 중단 |
| `published` | 홍보 노출 중 | 노출 기간 추적 |

## 영수증과 결제 내역

사용자는 결제 완료 후 이메일 영수증과 계정 내 결제 내역을 볼 수 있어야 한다.
세금계산서 또는 현금영수증 요구는 결제 제공자와 국내 운영 정책 검토 후 별도
구현 이슈에서 확정한다.

## 환불 원칙

- 사용자가 결제 직후 게시 전 취소하면 전액 환불 후보로 둔다.
- 검수 반려 시 사용자는 수정 후 재검수 또는 환불을 선택한다.
- 이미 노출이 시작된 홍보는 남은 기간 환불 여부를 운영 정책으로 정한다.
- 환불 처리와 노출 중단은 같은 관리자 액션 로그에 남긴다.

## 결제 제공자 선택 기준

| 기준 | 확인 항목 |
| --- | --- |
| 국내 결제 | 카드, 간편결제, 영수증 처리 |
| 개발 경험 | 테스트 결제, webhook, TypeScript SDK |
| 운영 | 환불 API, 관리자 콘솔, 실패 알림 |
| 확장 | 정기 결제 없이 단건 결제에 적합한지 |

Stripe와 Toss Payments는 후보로 두되, 실제 수수료와 계약 조건은 구현 직전
최신 문서로 확인한다.

## 후속 구현

- 결제 provider 결정 ADR
- webhook 검증과 결제 상태 동기화
- 결제 내역 API와 계정 화면
- 검수 반려와 환불 관리자 액션
