# AI Agent 협업 워크플로우

이 문서는 작업자와 AI Agent가 Gherkin 인수 시나리오를 기준으로 기능 개발,
문구 수정, 문서 갱신, 검증 보강을 함께 진행하기 위한 절차다.

## 원칙

- `docs/product/acceptance/*.feature`는 사용자가 검토할 핵심 인수 시나리오다.
- `docs/product/feature-map.md`는 현재 기능 영역과 지원 상태를 파악하는 지도다.
- AI Agent는 작업 전에 관련 시나리오를 확인하거나 갱신하고, 사용자는 작업 후
  이 시나리오를 기준으로 결과를 검토한다.
- 이슈와 PR은 변경 배경, 논의, 히스토리를 남기는 곳이다.
- 구현은 인수 시나리오의 기대 결과를 만족시키는 방향으로 진행한다.
- 시나리오 기준이 바뀌면 관련 이슈에 이유를 남기고 `.feature` 파일을 함께
  수정한다.
- 검증은 인수 시나리오의 기대 결과를 보호하는지 기준으로 선택한다.

## 작업 시작 전

1. 현재 브랜치와 워킹트리를 확인한다.
2. 관련 GitHub 이슈의 목표와 완료 조건을 읽는다.
3. `docs/product/feature-map.md`에서 관련 기능 영역과 현재 상태를 확인한다.
4. `docs/product/acceptance/`에서 관련 `.feature` 파일을 찾는다.
5. 관련 시나리오의 Given, When, Then과 예시가 작업 목표와 맞는지 확인한다.
6. `docs/product/acceptance/automation-map.json`에 연결된 시나리오라면 대응 테스트도
   함께 확인한다.
7. 관련 시나리오가 없고 사용자 체감 동작이 바뀌는 작업이면 작은 초안을 추가한다.
8. 사용자 흐름을 바꾸는 작업이면 이슈 본문이나 댓글에 변경 의도를 남긴다.

작업 시작 전 판단은 다음 중 하나로 정리한다.

- `인수 시나리오 영향 없음`: 내부 리팩터링이나 사용자 흐름이 바뀌지 않는 변경.
- `영향 시나리오: rest-to-note.feature`: 기존 시나리오 기준 안에서 진행하는 변경.
- `새 시나리오 필요`: 현재 `.feature` 파일로 설명하기 어려운 사용자 흐름.

## 작업 중

1. UI 문구, 상태 메시지, 오류 메시지가 사용자의 다음 행동을 알려주는지 확인한다.
2. 선택 상태, 입력 상태, 재생 상태처럼 사용자가 직접 체감하는 상태 전이가
   인수 시나리오와 맞는지 확인한다.
3. 구현 중 제품 기준이 바뀌면 코드를 먼저 확정하지 말고 관련 이슈 댓글에
   변경 이유를 남긴다.
4. 기준 변경이 확정되면 관련 `.feature` 파일을 함께 갱신한다.
5. 기존 검증 스크립트가 변경된 흐름을 보호하는지 확인한다.
6. 보호되지 않는 핵심 흐름이면 단위 테스트, MVP 검증, 패키징 검증, 수동 확인
   중 적절한 검증을 추가하거나 PR에 남긴다.
7. 자동화된 시나리오의 의미가 바뀌면 `.feature` 태그, `automation-map.json`,
   테스트명에 포함된 시나리오 ID를 함께 갱신한다.

## PR 전

1. `git diff`로 사용자 흐름, 문서, 테스트 변경을 함께 확인한다.
2. 관련 `.feature` 파일의 시나리오와 실제 동작이 어긋나지 않는지 확인한다.
3. 시나리오 기준을 바꿨다면 관련 이슈에 이유가 남아 있는지 확인한다.
4. 기능 상태나 관련 시나리오 링크가 바뀌었다면 `feature-map.md`를 갱신한다.
5. 자동화 맵에 연결된 시나리오를 수정했다면 `acceptance-traceability` 테스트가
   통과하는지 확인한다.
6. PR 본문에 영향받은 `.feature` 파일이나 `인수 시나리오 영향 없음`을 적는다.
7. 시나리오 변경 여부를 적는다.
8. 검증 결과를 적는다.

PR의 `인수 시나리오` 섹션은 다음 중 하나로 작성한다.

```md
## 인수 시나리오

- 영향 시나리오: docs/product/acceptance/rest-to-note.feature
- 시나리오 변경: 있음
- 변경 이유: #123에서 삭제 정책 기준을 변경하기로 결정했습니다.
```

또는:

```md
## 인수 시나리오

- 영향 시나리오: 인수 시나리오 영향 없음
- 시나리오 변경: 없음
- 변경 이유: 내부 리팩터링입니다.
```

## 머지 후

1. 닫힌 이슈가 상위 트래킹 이슈에 포함되어 있으면 체크박스를 갱신한다.
2. 후속 이슈가 필요하면 기존 이슈 댓글 또는 새 이슈로 남긴다.
3. 로컬 `main`을 원격과 맞춘다.
4. 다음 작업 후보를 고를 때 `docs/product/acceptance/`의 `@discussion` 시나리오와
   열린 이슈를 함께 확인한다.

## 검증 선택 기준

- 순수 문서나 템플릿 변경: `git diff --check`
- score-core, timeline, MusicXML, playback 의미 변경: `npm test`
- renderer layout, 선택 좌표, system 배치 변경: `npm run verify:mvp`
- 패키징, 배포 파일, 설치 앱 흐름 변경: `npm run verify:package`
- 앱 전체 빌드나 Electron/Vite 설정 변경: `npm run build`

여러 영역을 건드리면 해당 검증을 조합한다. 검증을 생략한 경우 PR에 이유를 적는다.

## 자동화 확장 후보

- PR 템플릿의 `영향 시나리오` 값이 비어 있으면 경고한다.
- `.feature` 파일이 바뀐 PR은 관련 이슈 링크를 요구한다.
- `src/score-core`, `src/io`, `src/playback` 변경 시 `npm test` 실행을 권장한다.
- `src/renderer`, `src/engraving` 변경 시 `npm run verify:mvp` 실행을 권장한다.
- `package.json`, `electron.vite.config.ts`, `.github/workflows` 변경 시 빌드 또는
  CI 영향 확인을 요구한다.
