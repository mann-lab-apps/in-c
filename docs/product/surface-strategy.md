# 웹·데스크탑·모바일 제품 표면 전략

작성일: 2026-07-12
1차 결정일: 2026-07-14

이 문서는 in C의 웹, 데스크탑 앱, 향후 모바일 표면이 어떤 역할을 맡을지 정리하기
위한 제품 표면 전략이다. GitHub issue #301의 1차 결정으로 이 문서를 기준 문서로
둔다.

## 기준

in C의 상위 정체성은 사보 앱이 아니라 클래식에 대해 모르는 것을 숨기지 않고,
각자가 들은 것과 듣지 못한 것을 솔직하게 나눌 수 있는 소통 공간이다.

제품은 두 문제를 함께 본다.

- 비전문가 그룹: 클래식 문화에 들어가기 어렵고, 관심이 깊어질수록 주변 사람과
  공유할 언어와 맥락이 부족해지는 문제.
- 전문가 그룹: 공연 홍보와 정보교류가 오프라인 관계망, 개인 SNS, 단체 채팅방,
  예매처 상세페이지에 흩어져 반복 수작업과 발견성 부족이 생기는 문제.

따라서 표면별 역할은 기능을 똑같이 복제하는 것이 아니라, 각 사용 맥락에서 이 두
문제를 가장 적은 운영 부담으로 풀 수 있는지로 판단한다.

## 결정해야 할 질문

- 웹은 in C의 기본 진입점이자 공개 공유 표면인가.
- 데스크탑 앱은 in C 브랜드 안에서 어떤 작업 표면을 맡을 것인가.
- 모바일 앱은 감상과 소통의 일상 표면인가, 웹/데스크탑과 같은 기능을 모두 제공해야
  하는가.
- Chromatics는 별도 브랜드인가, in C 안의 창작/악보 편집 표면 이름인가.
- `Open in Chromatics`는 데스크탑 앱, 웹 편집기, 모바일 앱에서 각각 어디로
  이어져야 하는가.

## 표면별 역할 초안

### 웹

웹은 현재 가장 자연스러운 in C의 기본 표면이다.

- Columns, Compositions, Community, 공연 배너 슬롯의
  공개 탐색 허브.
- 작품/악보 상세, 공연 배너 후보, 공개 커뮤니티 글의 기준 URL.
- 검색 유입과 공유 링크가 쌓이는 장기 자산.
- 비전문가가 질문을 만나고, 전문가의 활동이 작품/공연/사람 맥락에서 발견되는 장소.
- 장기적으로는 Figma처럼 악보를 브라우저에서 열고, 저장하고, 링크로 공유하며,
  점차 함께 편집할 수 있는 서비스형 편집 표면.

### 데스크탑 앱

데스크탑 앱은 깊은 작업과 파일 처리가 필요한 표면이다.

- Chromatics 기반 악보 편집, MusicXML 열기/저장, PDF 변환, 로컬 파일 관리.
- 웹 Compositions에서 연 악보를 수정하고 다시 공유 가능한 형태로 만드는 작업 공간.
- 오프라인 작업과 긴 편집 세션에 강점을 둔다.
- in C 웹의 작품 페이지, 악보 라이브러리, 커뮤니티 공유로 이어지는 얕은 연동을
  제공한다.

현재 결정:

- 사용자에게 모이는 브랜드는 `in C`로 둔다.
- Chromatics는 별도 트래픽 목적지가 아니라 in C 데스크탑 앱의 악보 편집 표면
  이름으로 다룬다.
- 사용자가 같은 악보와 작품을 다룬다고 느끼도록 용어, 파일, CTA 규칙은 웹과
  데스크탑에서 맞춘다.

### 모바일 앱

모바일 앱은 깊은 편집보다 감상 전후의 짧은 확인과 소통에 적합한 표면이다.

- Columns 읽기와 저장.
- 작품, 공연, Community 탐색.
- 공연 전 알림, 공연 후 짧은 감상 기록, 질문과 댓글.
- 커뮤니티 대화와 피드백.
- 악보는 복잡한 편집보다 보기, 저장, 공유, 데스크탑/웹 Chromatics로 보내기 흐름을
  우선한다.

모바일 앱은 공연장 이동 중, 공연 직전, 공연 후, 레슨/연습실에서 빠르게 확인하고
말을 남기는 사용 맥락을 우선한다.

## 1차 결정

1차 결정은 `in C 단일 브랜드 + 표면별 특화 역할`이다.

- 웹: in C의 공개 탐색과 공유 기준 표면.
- 데스크탑 앱: in C의 악보 편집, PDF 변환, 오프라인 작업 표면. Chromatics는 이
  작업 표면의 이름으로 쓰되, 장기적으로 웹 편집 표면과 같은 편집 경험을 공유한다.
- 모바일 앱: 감상, 공연 발견, 알림, 짧은 질문과 대화에 특화.

이 방향은 in C가 단순 사보 앱으로 오해되는 것을 줄이면서도, 이미 구현된 Chromatics
자산을 in C 안의 창작 도구로 살릴 수 있다. 트래픽과 기억은 in C로 모으고,
사용자가 배우는 규칙은 가능한 한 하나로 유지한다.

### 이름과 CTA 기준

- `in C`: 사용자에게 모이는 상위 브랜드와 제품 이름.
- `Chromatics`: in C 안의 악보 창작/편집 표면 이름. 별도 브랜드나 별도 유입
  목적지로 앞세우지 않는다.
- `Open in Chromatics`: 웹 Compositions 또는 작품 페이지에서 MusicXML/Chromatics
  원본을 여는 1차 CTA.
- 모바일 앱: 2026년 7월에는 구현하지 않고 감상, 공연 발견, 짧은 피드백, 질문/댓글
  중심 MVP로만 범위를 둔다.

### 기능 일치 기준

| 기능 | 웹 | 데스크탑 앱 | 모바일 |
| --- | --- | --- | --- |
| Columns 읽기 | 기준 표면 | 웹 링크로 연결 | MVP 후보 |
| Compositions 탐색 | 기준 표면 | 파일 열기 대상으로 연결 | 보기/저장 후보 |
| 악보 편집 | MVP 이후 점진 확장 | 기준 표면 | 보기/가벼운 수정 후보 |
| PDF 변환 | 안내 | 기준 표면 | 제외 |
| 공연 배너/Community | 기준 표면 | 얕은 홍보 링크 | MVP 후보 |
| 계정/커뮤니티 | Supabase/API 이후 | 얕은 상태 표시 | MVP 후보 |

### 혼동을 줄이는 공통 규칙

웹, 데스크탑, 모바일 표면은 역할이 다르지만 같은 in C 경험으로 느껴져야 한다.

- 같은 악보는 웹, Chromatics, 향후 모바일에서 같은 작품/악보 이름과 출처를 쓴다.
- `Chromatics에서 열기`는 항상 편집 가능한 악보 원본을 여는 행동이다. 단기에는
  데스크탑 앱으로, 장기에는 웹 편집기 또는 데스크탑 앱 중 가장 자연스러운 표면으로
  이어질 수 있다.
- `MusicXML 다운로드`는 항상 다른 사보 도구에서도 열 수 있는 fallback이다.
- `PDF 변환`은 Chromatics의 역할로 설명하고, 웹에서는 정적 PDF를 기본 행동으로
  앞세우지 않는다.
- 어떤 표면에서 기능을 지원하지 않으면 다른 의미로 바꾸지 않고, 지원하는 표면으로
  보내거나 준비 중임을 표시한다.
- 웹, 데스크탑, 모바일이 같은 기능을 모두 복제하지는 않지만, 같은 이름의 행동은
  같은 결과를 기대할 수 있어야 한다.

## Compositions와 Chromatics 흐름

현재 Compositions의 권장 흐름은 다음과 같다.

`웹에서 보기 -> Chromatics에서 열기 -> 필요하면 Chromatics에서 PDF 변환`

장기 서비스형 흐름은 다음과 같다.

`웹에서 보기 -> 브라우저에서 편집 -> 링크로 저장/공유 -> 필요하면 데스크탑 앱에서 이어 열기`

fallback 흐름은 다음과 같다.

`웹에서 보기 -> MusicXML 다운로드 -> Chromatics 또는 다른 도구에서 열기`

정적 PDF 파일을 기본 배포물로 앞세우지 않는다. PDF가 필요하면 사용자가 Chromatics
화면에서 직접 변환하는 흐름을 우선한다.

## 문서 작성 규칙

- in C 전체를 `악보 편집기`, `사보 앱`, `MuseScore 대체제`로 설명하지 않는다.
- Chromatics 문맥에서만 악보 편집기, 사보 도구, MusicXML, PDF 변환을 말한다.
- 웹, 데스크탑 앱, 모바일 역할은 위 1차 결정을 기본값으로 표현한다.
- 표면이 다르다고 해서 같은 행동 이름을 표면마다 다르게 쓰지 않는다.
- 구현되지 않은 모바일 앱 기능은 현재 기능처럼 쓰지 않는다.
- 제품 표면 전략은 최소 인원 운영 원칙과 함께 검토한다.
