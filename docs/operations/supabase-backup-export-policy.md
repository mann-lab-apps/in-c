# Supabase 백업·export 정책 초안

이 문서는 Supabase 운영을 시작하기 전에 합의할 최소 기준이다. 현재는 Supabase
프로젝트를 만들거나 데이터를 옮기지 않는다. 실제 적용은 [백엔드 계획](../product/supabase-backend-plan.md)의
승인 절차를 따른다.

## 인수 기준

- 공개 콘텐츠와 비공개 피드백의 백업을 분리한다.
- 각 데이터의 기준 원본, 실행 주기, 담당 역할, 보관 기준을 확인할 수 있다.
- 백업 파일을 만들기만 하지 않고 복구 확인 시점과 기록 항목을 정한다.
- 개인정보나 비밀 키를 공개 저장소에 넣지 않는다.
- 운영 시작 전에 보관 기간과 삭제 책임자를 다시 승인한다.

## 현재 단계

Supabase 전환 전 공개 콘텐츠의 기준 원본은 Git으로 관리하는 정적 파일이다. 자세한
파일 목록과 변경 절차는 [정적 콘텐츠 데이터 운영 규칙](static-content-source-of-truth.md)을
따른다. `feedback_events`는 설계 초안만 있으며 아직 운영 데이터 저장소가 아니다.

따라서 지금 할 일은 이 정책을 검토하는 것까지다. Supabase project 생성, 예약 작업,
외부 저장소 연결, 비밀 키 등록은 별도 승인 없이는 실행하지 않는다.

## 공개 콘텐츠

| 항목 | 운영 기준 |
| --- | --- |
| 대상 | `works`, `creators`, `concerts`, `classes`, `columns`, `scores`와 관계 데이터 |
| 기준 원본 | 전환 전에는 저장소의 정적 파일, 전환 후에는 승인된 Supabase 테이블 |
| export 주기 | 전환 후 매일 1회, 스키마 변경과 대량 수정 직전 추가 1회 |
| 형식 | 테이블별 CSV와 전체 복구용 database dump를 함께 보관 |
| 담당 역할 | 운영 담당자가 실행 결과를 확인하고, 개발 담당자가 복구 가능성을 확인 |
| 보관 초안 | 일별 30개와 월별 12개. 운영 시작 전에 비용과 변경량을 보고 확정 |

공개 데이터라도 draft 상태의 콘텐츠와 내부 메모는 공개 경로에 두지 않는다. export에
환경 변수, service role key, 접속 문자열이 포함되지 않았는지 보관 전에 확인한다.

## 비공개 피드백

| 항목 | 운영 기준 |
| --- | --- |
| 대상 | 공개 읽기가 금지된 `feedback_events` |
| 수집 범위 | `content_type`, `content_slug`, 미리 정한 `answer`, 생성 시각 등 승인된 값만 사용 |
| export 주기 | 운영 데이터가 생긴 뒤 매일 1회 |
| 보관 위치 | 접근 권한이 제한되고 저장 시 암호화되는 비공개 저장소 |
| 담당 역할 | 운영 담당자만 export 접근, 개발 담당자는 승인된 복구 점검 때만 접근 |
| 보관 초안 | 기간을 임의로 확정하지 않는다. 공개 전 개인정보 고지와 삭제 기준 승인 후 결정 |

자유 입력 원문과 이메일, 이름, 전화번호, 주소 같은 직접 식별자는 수집하거나 export에
추가하지 않는다. 이 기준은 [분석 이벤트 규칙](../product/analytics-events.md)과 공개
privacy 안내를 함께 따른다.

## 복구 확인

운영 시작 전 빈 검증 환경에서 최초 복구 연습을 한다. 이후에는 분기마다 1회, 스키마
변경 뒤 1회 실행한다. 실제 운영 DB를 덮어쓰지 않고 다음을 확인한다.

1. 가장 최근 dump를 새 검증 환경에 복구한다.
2. 공개 콘텐츠의 레코드 수와 주요 관계가 export 기록과 일치하는지 확인한다.
3. `feedback_events`를 익명 사용자로 읽을 수 없고 관리자 역할만 읽을 수 있는지 확인한다.
4. 표본 레코드의 ID, 상태, 생성 시각을 원본과 비교한다.
5. 실행 날짜, 사용한 백업, 결과, 확인자, 발견한 문제를 비공개 운영 기록에 남긴다.

복구가 실패하거나 최근 성공한 백업이 하루 이상 없으면 데이터 쓰기 전환을 멈추고
원인을 확인한다. 유효한 백업을 확인하기 전에는 오래된 백업을 삭제하지 않는다.

## 보관과 삭제

- 백업 접근 권한은 운영 담당자와 승인된 복구 확인자에게만 준다.
- 담당자가 바뀌면 이전 권한을 제거하고 접근 목록을 다시 확인한다.
- 보관 기간이 끝난 파일은 복구 성공 기록을 확인한 뒤 삭제한다.
- 사용자 삭제 요청이 적용되는 데이터가 생기면 운영 DB뿐 아니라 백업에서의 만료
  시점과 예외를 개인정보 고지에 먼저 적는다.
- 법률 또는 계약상 별도 보관 의무가 생기면 이 초안을 그대로 적용하지 않고 재검토한다.

## 운영 시작 승인표

- [ ] Supabase project와 운영 담당자가 승인되었다.
- [ ] 공개 콘텐츠와 피드백의 최종 보관 기간이 승인되었다.
- [ ] 비공개 백업 위치와 접근 권한이 확인되었다.
- [ ] 자동 export 실패 알림을 받을 담당자가 정해졌다.
- [ ] 최초 복구 연습이 성공했고 기록 위치가 정해졌다.
- [ ] privacy 안내와 실제 수집 필드가 일치한다.

하나라도 확인되지 않으면 Supabase를 데이터 기준 원본으로 전환하지 않는다.
