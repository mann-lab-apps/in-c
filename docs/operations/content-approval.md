# 콘텐츠 검수와 공개 승인 기준

이 문서는 정적 데이터로 관리하는 공개 콘텐츠의 최소 운영 기준이다. 실제 계정,
Supabase 설정, 배포 권한은 다루지 않는다.

## 인수 기준

- 콘텐츠 작성자, 검수자, 공개 담당자의 책임과 하지 않는 일을 구분한다.
- GitHub issue에서 시작해 PR 검수와 공개 결정으로 끝나는 흐름을 설명한다.
- 같은 사람이 여러 역할을 맡을 때도 별도 검수 기록을 남기는 조건을 설명한다.
- 출처, 저작권, 개인정보 또는 검증 결과가 불확실하면 공개하지 않는다고 명시한다.
- 기존 Compositions 후보 흐름과 PR 양식을 바로 찾을 수 있게 연결한다.

## 실패 가능한 검증 기준

- `작성자`, `검수자`, `공개 담당자` 중 하나라도 책임이 없으면 실패한다.
- `GitHub issue`, `PR`, 승인, 공개 순서 중 하나라도 빠지면 실패한다.
- 공개를 멈춰야 하는 조건이 없으면 실패한다.
- 관련 문서의 상대 링크가 존재하지 않으면 실패한다.
- `git diff --check`가 실패하면 완료로 보지 않는다.

## 역할

| 역할 | 책임 | 하지 않는 일 |
| --- | --- | --- |
| 작성자 | issue에 출처와 변경 목적을 남기고 콘텐츠와 검증 결과를 PR로 제출한다. | 자신의 제출만으로 공개 승인을 확정하지 않는다. |
| 검수자 | 출처, 저작권, 개인정보, 문구 정확성, 검증 결과를 확인하고 PR에 승인 또는 보완 요청을 남긴다. | 불확실한 항목을 추정으로 승인하지 않는다. |
| 공개 담당자 | 검수 기록과 필수 검증 통과를 확인한 뒤 공개 반영 여부를 결정한다. | 검수 기록이 없는 콘텐츠를 공개 상태로 바꾸지 않는다. |

한 사람이 여러 역할을 맡을 수 있다. 다만 작성자와 검수자가 같다면 PR 본문의
체크리스트에 확인 근거를 남기고, 공개 담당자가 그 기록을 다시 확인한다. 별도 공개
담당자가 없다면 공개를 보류하고 다른 검수자를 지정한다.

## 승인 흐름

1. 작성자가 콘텐츠별 GitHub issue를 만들고 목적, 출처, 공개 범위, 관련 이슈를 적는다.
2. 작성자가 issue와 연결된 작업 브랜치와 PR을 만들고 변경 내용과 검증 결과를 적는다.
3. 검수자가 PR의 출처, 권리, 개인정보, 사용자 문구와 검증 결과를 확인한다.
4. 보완이 필요하면 검수자가 PR에 이유를 남기고 작성자가 수정한다.
5. 검수자가 승인 기록을 남긴 뒤 공개 담당자가 공개 조건을 다시 확인한다.
6. 공개 담당자가 승인한 PR만 정해진 브랜치에 반영한다. 공개 카탈로그의 상태 변경은
   이 반영과 함께 처리한다.

issue는 작업 배경과 판단 기록의 기준이고, PR은 실제 변경과 검증 기록의 기준이다.
말이나 별도 메신저로 합의했더라도 공개 판단은 issue 또는 PR에 짧게 다시 남긴다.

## 공개 중단 조건

다음 중 하나라도 해당하면 PR을 공개 상태로 반영하지 않는다.

- 원자료의 출처나 사용 권리를 확인할 수 없다.
- 현대 편곡, 채보, 번역, 해설을 복제했는지 판단하기 어렵다.
- 공개하면 안 되는 개인정보나 비공개 자료가 포함되어 있다.
- 필수 metadata, 사용자에게 보이는 출처 설명 또는 관련 파일이 빠졌다.
- PR의 관련 검증이 실패했거나 결과가 기록되지 않았다.
- 검수자 승인 또는 공개 담당자의 최종 확인이 없다.

중단 이유와 필요한 다음 행동을 PR에 남긴다. 해결 전 콘텐츠는 issue에만 두며 공개
카탈로그나 사용자 화면에는 넣지 않는다.
