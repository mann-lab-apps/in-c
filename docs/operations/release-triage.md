# 릴리즈 이슈 분류 기준

이 문서는 릴리즈 전에 열린 이슈를 빠르게 분류하는 기준이다. 기능의 중요도가
아니라, 현재 릴리즈를 내보내도 되는지를 판단한다.

## 5분 판단 순서

1. 재현 조건과 영향을 확인한다.
2. 아래 `release blocker` 기준에 하나라도 해당하는지 확인한다.
3. 실제 문제에는 `bug`, 개선 제안에는 `enhancement`를 붙인다.
4. 7일 안에 해결해야 하는 blocker에만 `긴급`을 추가한다.
5. 중복 또는 이미 알려진 문제인지 확인하고 판단 근거를 이슈에 남긴다.

## Release blocker 판정

다음 중 하나라도 해당하면 현재 릴리즈를 보류한다.

- 저장하거나 다시 여는 과정에서 사용자 데이터가 사라지거나 달라진다.
- 앱이 시작되지 않거나 핵심 흐름에서 반복해서 종료된다.
- 지원한다고 안내한 설치, 악보 편집, 저장, 열기 흐름을 완료할 수 없다.
- `moderate` 이상 보안 검증이 실패했거나 secret 노출 가능성이 확인됐다.
- 필수 CI 검증이 실패했고 원인이 설명되지 않았다.
- 프로덕션 URL, 다운로드 파일 또는 배포가 사용자에게 열리지 않는다.

문구, 작은 배치 어긋남, 문서 누락처럼 우회할 수 있고 데이터와 핵심 흐름에
영향이 없는 문제는 기본적으로 blocker가 아니다. 제한사항 또는 릴리즈 노트에
알리고 후속 이슈로 추적한다.

판단이 애매하면 blocker로 단정하지 않는다. 재현 조건과 영향 범위를 보강하고
릴리즈 담당자가 다시 판단하도록 `판정 보류`로 기록한다.

## 라벨 사용

| 라벨 | 붙이는 경우 | 붙이지 않는 경우 |
| --- | --- | --- |
| `긴급` | release blocker이며 7일 안에 처리해야 할 때 | 중요하지만 현재 릴리즈를 막지 않을 때 |
| `bug` | 기대한 동작과 실제 동작이 다르고 재현할 수 있을 때 | 새 기능이나 문서 개선 제안일 때 |
| `enhancement` | 새 기능, 사용성 개선, 콘텐츠 보강 제안일 때 | 기존 동작이 깨진 문제일 때 |
| `documentation` | 문서만 추가하거나 고칠 때 | 코드 동작 수정이 핵심일 때 |
| `duplicate` | 같은 원인과 완료 조건을 다루는 대표 이슈가 있을 때 | 비슷해 보여도 원인이나 완료 조건이 다를 때 |

`긴급`은 문제 종류가 아니므로 `bug` 또는 필요한 다른 라벨과 함께 쓴다.
라벨이 맞지 않으면 새 라벨을 만들지 말고 이슈 본문에 판단을 먼저 기록한다.

## 중복과 이미 알려진 문제

- 중복 이슈는 대표 이슈 번호와 같은 범위인 이유를 댓글로 남기고
  `duplicate`를 붙인 뒤 닫는다.
- 원인이나 완료 조건이 다르면 중복으로 닫지 않고 관련 이슈로 연결한다.
- 이미 알려진 문제는 `known limitation` 문서나 릴리즈 노트의 위치를 연결한다.
- 알려진 문제라도 영향이 커졌거나 우회 방법이 사라졌으면 blocker 여부를 다시
  판단한다.
- 보안 또는 개인정보 내용은 공개 이슈에 민감한 재현 정보를 남기지 않는다.

## 이슈 기록 양식

```markdown
## 릴리즈 판정

- 판정: blocker / blocker 아님 / 판정 보류
- 사용자 영향:
- 재현 조건:
- 근거가 된 검증:
- 적용 라벨:
- 중복 또는 known issue:
- 다음 행동과 담당자:
```

릴리즈 전체의 최종 판정은
[`Release Readiness Checklist`](../quality/release-readiness-checklist.md)를 따른다.
프로덕션 장애 대응과 롤백은
[`프로덕션 운영 플레이북`](production-playbook.md)을 따른다.
