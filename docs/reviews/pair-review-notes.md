# 페어 전체 점검 기록

이 문서는 빠른 CI 머지 이후 사용자가 개별 PR diff를 모두 보지 않는 운영에서, 페어로
전체 상태를 훑으며 실제로 어떤 질문을 하게 되는지 관찰하기 위한 기록장이다.

아직 체크리스트가 아니다. 같은 질문이 여러 번 반복될 때만 release routine, snapshot
초안, QA 체크리스트로 승격한다.

## 기록 원칙

- 정리된 답보다 실제로 나온 질문을 먼저 남긴다.
- 코드 diff보다 앱, 사이트, 문서, 이슈, 릴리즈 가능성 같은 표면을 본다.
- 사용자가 불안하다고 느낀 지점은 근거가 부족해도 남긴다.
- 한 번만 나온 질문은 바로 루틴으로 만들지 않는다.
- 반복해서 나온 질문만 `반복 후보`로 표시한다.

## 기록 템플릿

```md
## YYYY-MM-DD 페어 전체 점검

### 기준

- 기준 branch/commit:
- 최근 태그:
- 확인한 PR/이슈:
- 같이 본 표면:

### 사용자가 실제로 물어본 것

- [ ] [질문]

### 괜찮아 보인 것

- [메모]

### 불안하거나 직접 확인하고 싶은 것

- [메모]

### 문서화 완료지만 제품 반영이 남아 보이는 것

- [메모]

### 후속 이슈 후보

- [메모]

### 반복 후보

- [ ] 이 질문은 다음 점검에서도 다시 나올 것 같다: [질문]

### 루틴으로 승격하지 않을 것

- [메모]
```

### 현재 관찰 가설

- 사용자는 개별 PR diff보다 전체 제품 상태와 릴리즈 가능성을 보고 싶어 한다.
- `snapshot-template.md`는 아직 active 루틴으로 쓰기보다 Pending 초안으로 두는 편이 낫다.
- 페어 점검에서 반복되는 질문을 먼저 모은 뒤 문서화하는 방식이 더 자연스럽다.

### 첫 점검에서 열어볼 표면 후보

- 앱/Chromatics 첫 사용 흐름

- `docs/quality/known-limitations.md`
- 열린 이슈 중 release blocker 또는 사용자 결정 필요 항목
- 최근 머지 PR 중 사용자-facing 변경
